Documentation du diagramme

  • Perimetre volontairement simplifie: le diagramme montre 10 cas metier lisibles, sans detail interne de calcul ou de signature.
  • Roles internes: Administrateur gere configuration et conformite, Comptable porte le flux complet de facturation, Caissier se limite a emission/consultation, Auditeur lit archives et rapports.
  • Roles externes: DGI/SECeF n'intervient que sur la transmission et le retour de statut; Client intervient uniquement pour la verification QR.
  • Dependances essentielles: UC4 emettre facture certifiee produit UC5 transmission; UC5 mene vers UC6 uniquement en cas de rejet.
  • Traçabilite explicite: UC4 et UC7 alimentent UC8 afin de garantir la preuve d'audit sans surcharger le diagramme.

Documentation du diagramme

  • Centre du modele: Facture est l'entite pivot. Toutes les preuves de conformite sont reliees a Facture (lignes, taxes, scellement, transmission, audit, avoir).
  • Cardinalites a respecter: 1 Facture -> n LigneFacture, 1 Facture -> n TransmissionSecef, 1 Facture -> n JournalAudit, 1 Facture -> 0..n Avoir.
  • Numerotation: SerieFacture.prochainNumero est la source unique de la sequence. Le numero ne doit jamais etre recycle.
  • Integrite cryptographique: ScellementNumerique est 0..1 sur Facture, car une facture certifiee porte un scellement unique pour une version donnee.
  • Traçabilite fiscale: TransmissionSecef conserve format, hash, code retour, message et compteur de retries pour reconstituer l'historique exact.

3) Diagramme de sequence

sequenceDiagram autonumber actor U as Utilisateur participant FE as Frontend Vue.js participant API as Backend Spring Boot participant DB as PostgreSQL participant SEC as DGI SECeF U->>FE: Saisir facture FE->>API: POST /factures API->>DB: Verifications + calculs + numerotation API->>DB: Enregistrer facture + audit API->>SEC: Envoyer payload FEC SEC-->>API: ACK acceptee/rejetee API->>DB: MAJ statut transmission API-->>FE: Retour numero + statut FE-->>U: Affichage facture

Documentation du diagramme

  • Chronologie exacte: U->FE saisie, FE->API creation, API->DB validations et numerotation, API->SEC transmission, SEC->API ACK, API->DB mise a jour, API->FE resultat.
  • Point de controle critique: l'enregistrement facture + audit est fait avant l'envoi SECeF pour garantir la preuve locale meme en cas de panne reseau.
  • ACK traite: le retour SECeF doit mettre a jour le statut de transmission et le statut metier de la facture.
  • Idempotence: la reprise d'envoi doit reutiliser les identifiants de facture sans creer de doublon fonctionnel.
  • Sortie front: le frontend affiche au minimum numero facture, statut transmission et message retour en cas de rejet.

4) Diagramme de flux

flowchart TD A[Creer facture] --> B[Valider donnees] B --> C[Calculer HT TVA TTC] C --> D[Generer numero sequentiel] D --> E[Generer code auth] E --> F[Generer QR] F --> G[Certifier / sceller] G --> H[Enregistrer + audit] H --> I[Transmettre au SECeF] I --> J{ACK recu ?} J -- Non --> K[Retry] K --> I J -- Oui --> L{Statut} L -- Accepte --> M[Archiver] L -- Rejete --> N[Corriger + retransmettre] N --> I

Documentation du diagramme

  • Etapes obligatoires: B, C, D, E, F, G et H sont sequentielles et non sautables pour respecter la conformite.
  • Noeud J ACK recu?: condition purement technique (transport et disponibilite plateforme).
  • Noeud L Statut: condition fonctionnelle retour DGI (accepte vs rejete).
  • Branche retry K: ne modifie pas le contenu facture, elle rejoue uniquement la transmission.
  • Branche correction N: modifie les donnees metier, puis necessite une nouvelle certification avant envoi.

5) Diagramme d'etats

stateDiagram-v2 [*] --> BROUILLON BROUILLON --> CERTIFIEE: Validation + scellement CERTIFIEE --> ENVOYEE: Transmission SECeF ENVOYEE --> ACCEPTEE: ACK accepte ENVOYEE --> REJETEE: ACK rejete REJETEE --> CERTIFIEE: Correction + recertification ACCEPTEE --> ANNULEE: Avoir

Documentation du diagramme

  • BROUILLON -> CERTIFIEE: transition autorisee uniquement si validations metier et fiscales sont conformes.
  • CERTIFIEE -> ENVOYEE: transition creee au moment de l'envoi SECeF, avec trace de tentative.
  • ENVOYEE -> ACCEPTEE: ACK positif DGI, facture fiscalement validee.
  • ENVOYEE -> REJETEE: ACK negatif DGI avec motif obligatoire.
  • REJETEE -> CERTIFIEE: possible seulement apres correction metier et recertification complete.
  • ACCEPTEE -> ANNULEE: uniquement via avoir; la facture initiale reste archivee.

6) Diagramme de composants

flowchart LR UI[Frontend Vue.js] --> API[Backend Spring Boot] API --> FACTURE[Module Facturation] API --> CONFORMITE[Module Conformite] API --> TRANSMISSION[Module Transmission SECeF] API --> REPORTING[Module Reporting] FACTURE --> DB[(PostgreSQL)] CONFORMITE --> DB TRANSMISSION --> DB TRANSMISSION --> SECEF[(API DGI SECeF)]

Documentation du diagramme

  • Frontend Vue.js: collecte les donnees et ne calcule pas la conformite fiscale finale.
  • Module Facturation: cree facture et lignes, calcule bases HT/TVA/TTC.
  • Module Conformite: applique numerotation, code auth, QR, scellement et audit.
  • Module Transmission: transforme en payload DGI, envoie, recoit ACK, gere retries.
  • Module Reporting: produit rapports periodiques pour controle fiscal.

7) Diagramme de packages

flowchart TB C1[api.controller] --> C2[application.service] C2 --> C3[domain.model] C2 --> C4[domain.repository] C4 --> C5[infrastructure.persistence] C2 --> C6[infrastructure.secef] C1 --> C7[infrastructure.security] C2 --> C8[infrastructure.audit] C2 --> C9[shared]

Documentation du diagramme

  • api.controller: convertit HTTP en commandes applicatives, sans logique metier.
  • application.service: orchestre les cas d'usage (creer facture, certifier, transmettre, corriger).
  • domain.model: contient entites, enums et invariants.
  • domain.repository: contrats abstraits, utilises par les services.
  • infrastructure.*: implementations techniques (JPA, SECeF, securite, audit).

8) Diagramme de deploiement

flowchart LR BROWSER[Navigateur] --> LB[Reverse Proxy] LB --> WEB[Frontend Vue.js Nginx] LB --> API1[Spring Boot API 1] LB --> API2[Spring Boot API 2] API1 --> PG[(PostgreSQL)] API2 --> PG API1 --> WORKER[Worker Retry] API2 --> WORKER WORKER --> SECEF[(SECeF DGI)]

Documentation du diagramme

  • Trajet utilisateur: Navigateur -> Reverse Proxy -> Frontend/API.
  • Scalabilite API: API1 et API2 servent les memes endpoints et la meme base PostgreSQL.
  • Traitements differes: Worker execute retries et envois asynchrones vers SECeF.
  • Couplage minimal: le frontend ne contacte jamais directement la DGI.
  • Securite reseau: chiffrement TLS, filtrage au proxy, journalisation des appels sortants SECeF.
